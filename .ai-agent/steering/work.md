# 開発ワークフロー

## 基本原則

- ユーザーとの対話的な進行: 各ステップでユーザーの確認を取る
- 早期フィードバック: 動く状態を早く作り、確認を得る
- 小さな積み重ね: 小さな単位で実装し、動作確認を繰り返す
- **依存最小**: Gemini CLI 以外の外部依存を増やす変更には慎重になる

## ブランチ戦略（GitHub Flow）

本プロジェクトでは GitHub Flow を採用する。

### ブランチの種類

- **デフォルトブランチ**（`main`）: 常にデプロイ可能な状態を維持する。直接コミットしない
- **作業ブランチ**: タスクごとにデフォルトブランチから作成する。命名規則:
  - 形式: `<type>/<kebab-case-description>`
  - 小文字の英数字とハイフンを使用
  - type の種類:
    - `feature/` — 新機能・改善・バグ修正（例: `feature/add-prompt-tool`）
    - `support/` — メンテナンス・ドキュメント・依存関係更新など（例: `support/update-dependencies`）

### フロー

```
main ─── feature ブランチ作成 ─── 実装・コミット ─── PR 作成 ─── レビュー ─── マージ ─── main に戻る
```

1. **ブランチ作成**: `main` から `git checkout -b {ブランチ名}` で作成
2. **実装・コミット**: 小さな単位でコミットを積み重ねる
3. **PR 作成**: 実装完了後、`/autodev-create-pr` で PR を作成
4. **レビュー・マージ**: レビューを経て `main` にマージ
5. **ブランチ切り替え**: マージ後、`/autodev-switch-to-default` でデフォルトブランチに戻る

### レビュー形式

本プロジェクトは **GitHub レビュー**を採用する。

1. `/autodev-review-pr` で reviewer エージェントが GitHub にレビューコメントを投稿
2. `/autodev-import-review-suggestions` で GitHub 上のレビューコメントを取り込み、対話的に修正

### 注意事項

- `main` への直接プッシュは避ける
- マージ方式はマージコミット（`merge commit`）を使用する
- マージ済みのフィーチャーブランチはリモートで自動削除されることを前提とする

## タスク実行フロー

### 着手前

1. タスクの README を作成（目的、方針、完了条件、作業ログ）
2. 関連する steering ドキュメント（特に `product.md` / `tech.md`）を確認
3. 方針をユーザーに確認
4. フィーチャーブランチを作成

### 実装中

- ステップごとに動作確認
- lint / 型チェック / テストを実行して品質を維持
- 意味のある単位でコミット（コミットメッセージは変更内容を簡潔に記述）
- 必要に応じてユーザーにフィードバックを求める

### 動作確認コマンド（暫定）

実装が進んだ段階で以下を整備し、ここに反映する:

```bash
npm run typecheck   # 型チェック
npm run lint        # lint
npm test            # ユニットテスト
npm run build       # ビルド
```

DXT パッケージ動作確認は、ビルドした `.dxt` を Claude Desktop にインストールして手動検証する。

### 完了時

1. 完了条件を満たしているか確認
2. 作業ログを更新
3. PR を作成（`/autodev-create-pr`）
4. ユーザーに完了報告

## プロダクト固有の注意

- **Gemini API SDK を直接導入しない** — Gemini CLI 経由で完結させる方針を維持する。直接 SDK を
  使う変更を提案する場合は、必ず `product.md` の非スコープ項目との整合を確認する
- **Workspace 認証フローを壊さない** — Gemini CLI 側の OAuth 状態に依存する設計のため、認証関連
  の変更は慎重に。手元で `gemini auth login` 済みの状態でテストする
- **DXT 同梱サイズに注意** — 依存追加時は最終的な `.dxt` サイズへの影響を意識する
